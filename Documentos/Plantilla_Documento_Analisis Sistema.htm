2. DESCRIPCIÓN DEL SISTEMA

UNIDMER, es un sistema de información de pedidos desarrollado para Laboratorios Merey en el cual se pretende agilizar el proceso de toma y transferencia de pedidos, aspectos mencionados en documentos anteriormente presentados; el sistema estará regido por los siguientes módulos:

GESTION DE PEDIDOS Y GESTION DE INVENTARIOS, en estos dos módulos se verá reflejada la actividad de ventas del laboratorio. Módulos que serán utilizados por todos los actores que harán uso del sistema.

4. CASOS DE USO

4.1 Casos de uso del módulo Gestión de usuarios

En este paquete como su mismo nombre lo indica, se quiere hacer el manejo de los usuarios junto con las interacciones que estos tienen dentro del sistema, ya que en este módulo solo se almacenará las entradas, salidas y modificaciones de los actores. Claramente habrá actividades que no todos los usuarios podrán realizar como por ejemplo crear usuarios ya que es labor única del administrador. En estos casos el sistema identificará automáticamente las actividades que puede realizar x o y rol.

 

 

 

 

 

 

 

 

 

4.2. Casos de uso del módulo Gestión de novedades

En el módulo gestión de novedades se encargará de manejar las novedades que el

administrador considere pertinentes para que el resto de usuarios existentes en el

sistema conozcan; se agregó el caso de uso de visualizar catalogo debido a que el

Laboratorio constantemente actualiza la imagen de sus productos, precios y existencias.

 

 

 

 

 

 

 

 

4.. 3 Casos de uso del módulo Gestión de reportes

 

 

En este paquete se reflejarán los reportes de la actividad que desarrollen los diferentes

objetos dentro del sistema, ya que se llevará un control sobre los pedidos y el inventario

 que mantengan el sistema de ventas del laboratorio; este módulo solo podrá ser

 manejado por el usuario administrador.

4.. 4 Casos de uso del módulo Gestión de rutas

Este módulo será diseñado para que el usuario transferencista a manera de proceso de

venta gestione las rutas que harán parte de este proceso.

 

4.. 4 Casos de uso del módulo Gestión de pedidos

 

 

Este paquete maneja la información referente a la información de pedidos ya que en el

 interactúan los roles distribuidores, cliente y transferencista. Quienes son los que

modifican a la entidad pedidos

4.. 4 Casos de uso del módulo Gestión de inventario

 

Este módulo mostrará la interacción que hay entre los actores y la entidad productos, este

solo podrá ser usado por los usuarios distribuidor y administrador

6. MODELO DE CONCEPTOS DEL SISTEMA

6.1.1 Responsabilidades de la clase USUARIOS

Esta clase hereda atributos a las otras clases pertenecientes a los usuarios, junto con los métodos

6.1.2 Responsabilidades de la clase ADMINISTRADOR

Esta clase muestra al usuario administrador quien se va a encargar de procesos

anteriormente mencionados.

6.1.3 Responsabilidades de la clase TRANSFERENCISTA

Esta clase muestra al usuario transferencista que es quien se encargara de gestionar los pedidos

6.1.4 Responsabilidades de la clase CLIENTE

Esta clase muestra al usuario cliente que es aquel que se encarga de crear su pedido y realizar sus

actividades de usuario personales

 

6.1.5 Responsabilidades de la clase DISTRIBUIDOR

Este usuario se encargará de solicitar mercancía y descargar pedidos registrados por el transferencista

6.1.6 Responsabilidades de la clase PEDIDO

Esta clase consiste en el registro del envio del mismo, hace uso de la clase productos ya que un pedido

 está compuesto por productos

6.1.7 Responsabilidades de la clase PRODUCTOS

En esta clase se debe registrar, modificar y eliminar los productos pertenecientes al laboratorio

6.1.8 Responsabilidades de la clase PRE- FACTURA

Esta clase depende del pedido ya que sin la esta clase no existiría, además será generada por el

sistema cuando se complete correctamente el proceso de registro del pedido
